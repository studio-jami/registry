The `@jami-studio/harness-cli` package exposes the `jami` command for local Harness
inspection and evidence-oriented workflows.

```bash
npm install @jami-studio/harness-cli
jami doctor --json
```

Use `--json` for machine-readable output. The CLI writes local state under `.jami-harness`
in the current workspace.

## Commands

```bash
jami <init|run|resume|approve|deny|cancel|retry|inspect|doctor|tools|memory|context|docs|map|workbench|release|verify|migration> [--json]
```

| Command | Purpose |
| --- | --- |
| `init` | Initializes local Harness state. |
| `run` | Executes the local deterministic run path and produces evidence-oriented output. |
| `resume` | Reads checkpoint replay state. |
| `approve` / `deny` | Records approval decisions for policy-gated work. |
| `cancel` / `retry` | Fail-closed control commands for surfaces not implemented as runtime orchestration yet. |
| `inspect` | Reports modules, install paths, source locks, tool adapters, and unsupported surfaces. |
| `doctor` | Diagnoses setup and returns exact next steps. |
| `tools` | Lists local tool adapter capabilities and source locks. |
| `memory` / `context` | Inspects memory and context assembly capabilities. |
| `docs` | Reports repo-level docs generation status. |
| `map` | Prints the local module map. |
| `workbench` | Reports local static workbench support and hosted-workbench gaps. |
| `release` | Reports public package/release evidence and unsupported hosted routes. |
| `verify` | Checks local CLI state and core module availability. |
| `migration` | Fail-closed migration placeholder; checkpoint/store migrations are not implemented. |

## Common Checks

```bash
jami inspect --json
jami doctor --json
jami run --json
jami release --json
```

<Note>
  The CLI is local-first. Hosted provider/store/workbench/control-plane routes remain explicit
  operational lanes and are not implied by a successful local CLI run.
</Note>

## Package Boundary

The CLI depends on `@jami-studio/harness-sdk` and `@jami-studio/harness-store-local`.
Application code that embeds Harness should usually start with the SDK; operators and release
checks usually start with the CLI.
