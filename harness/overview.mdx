Jami Harness is the contract-first agent runtime foundation in the `@jami-studio/*` family. It owns the governed action loop: runs, tasks, tools, policies, approvals, artifacts, memory, traces, evidence, checkpoints, release packets, and developer-facing CLI/SDK surfaces.

Studio UI owns the token system, primitive vocabulary, registry items, workbench controls, and resident rendering layer. Harness and Studio UI stay separate repos and packages, but they meet through typed payloads and references.

<Note>
  The Harness executes and governs agent work. Studio UI displays trusted state and
  installable components. Registry items can show action refs and artifact refs,
  but they do not execute actions directly.
</Note>

## What the Harness owns

The Harness owns the product grammar that makes agent work reliable:

| Surface | Harness responsibility |
|---|---|
| Runs and tasks | Durable run ids, actor/project/environment context, retry and recovery state |
| Tools | Typed tool calls, permissions, cancellation, replay awareness, and audit records |
| Policy | Identity, scopes, approvals, denials, consent boundaries, and escalation |
| Memory and context | Project memory, task memory, citations, freshness, redaction, and retention |
| Artifacts | Evidence packets, generated reports, patches, screenshots, diagrams, and release outputs |
| Observability | OpenTelemetry-compatible events, traces, metrics, evals, and replay evidence |
| Developer surfaces | CLI, SDK, local workbench, examples, generated docs, and package manifests |

## How it fits with Studio UI

Harness-to-UI integration is data-first. The Harness emits typed references such as `uiPayload`, `artifactView`, `actionRef`, `themeRef`, and `suiteRef`. Studio UI validates and renders those references through resident allowlisted components.

Policy decisions, tool side effects, memory writes, provenance, trace emission, and runtime state remain Harness responsibilities. UI packages render state and configuration without taking ownership of execution.

## Modularity model

Harness capabilities fall into four classes:

- **Core invariants**: contracts, runtime lifecycle, policy seam, tool wrapper, artifact/evidence model, and observability event contract.
- **Included defaults**: local store, default memory, default context assembler, default policy engine, common provider/tool adapters, CLI, and SDK helpers.
- **Replaceable modules**: memory, context, providers, policy engine, artifact storage, trace/audit/metric sinks, secret resolver, hosted store, docs publishing output, and workbench shell.
- **Optional surfaces**: hosted dashboard, docs site, advanced eval packs, marketplace catalogs, SaaS control plane, cloud recipes, and Studio UI-powered workbench.

## Current public release posture

The first public package lane is published under the `@jami-studio` npm namespace. The registry docs track release evidence separately from hosted runtime claims: packages and release artifacts are public, while hosted provider/store/OTLP/control runtime lanes are still separate operational work.
