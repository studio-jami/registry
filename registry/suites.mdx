A **suite** is a bundled, opinionated application shell for a specific workflow lane. Installing one suite entry-point item triggers a complete install graph that includes the factory theme, all required UI primitives and components, every page and block descriptor for the lane, and the app shell implementation evidence. You get a working multi-route application scaffold without manually assembling individual items.

The registry ships four suites, each targeting a distinct type of application:

## Suite structure

Each suite is a `registry:suite` item. Its `registryDependencies` field defines the complete install graph ŌĆö the CLI walks this graph and installs every item in the correct order.

A suite manifest (`*.suite.json`) describes:

```json
{
  "name": "@jami-studio/solo-suite",
  "lane": "solo",
  "items": ["jami-theme", "button", "panel", "text-field", "data-list", "agent-panel", "docs-source-panel",
            "solo-app", "solo-today-page", "solo-docs-page",
            "solo-agenda-block", "solo-task-queue-block", "solo-agent-status-block",
            "solo-docs-sources-block", "solo-forms-block"],
  "installGraph": {
    "root": "solo-suite",
    "dependencies": [ ... ]
  },
  "plannedSurfaces": ["calendar", "forms", "docs", "agent", "tasks", "crm", "content", "admin"],
  "shell": { ... },
  "implementation": { ... }
}
```

| Field | What it tells you |
|---|---|
| `items` | Ordered list of every item the suite installs |
| `installGraph` | Dependency graph rooted at the suite entry-point |
| `plannedSurfaces` | Application surfaces planned for the lane (not all are implemented yet) |
| `shell` | Navigation routes, page definitions, block definitions, and component parts |
| `implementation` | Implementation evidence ŌĆö mounted React app, page, and block JSON file paths |

A compact personal-operations shell for calendar, tasks, docs, forms, CRM, content, admin, and display-only agent status.


## The install graph

When you install a suite, the shadcn CLI resolves `registryDependencies` recursively. The graph is rooted at the suite item and expands depth-first. Consider the `solo-suite` install order:

<Steps>
  <Step title="Theme layer">
    `jami-theme` ŌĆö CSS variables, Tailwind `@theme`, TypeScript token constants, and shadcn cssVars are written to `studio-ui/`.
  </Step>
  <Step title="Primitive layer">
    `button`, `panel`, `text-field` ŌĆö vocabulary, styles, descriptor, component factory, and Radix React wrappers are written to `components/ui/`.
  </Step>
  <Step title="Component layer">
    `agent-panel`, `docs-source-panel`, `data-list` ŌĆö additional component factory entries for the composed resident components.
  </Step>
  <Step title="Block layer">
    Each block item (e.g. `solo-agenda-block`) writes a pair of JSON files to `studio-ui/suites/solo/blocks/` ŌĆö a block descriptor and a block implementation descriptor with rendered state evidence.
  </Step>
  <Step title="Page layer">
    Each page item (e.g. `solo-today-page`) writes a page descriptor and a page implementation descriptor to `studio-ui/suites/solo/pages/`.
  </Step>
  <Step title="App layer">
    `solo-app` writes an app implementation descriptor to `studio-ui/suites/solo/` containing the full route map, navigation, and evidence summary.
  </Step>
  <Step title="Suite manifest">
    `solo-suite` writes the final suite manifest to `studio-ui/suites/solo.suite.json`, which records the complete item list, install graph, planned surfaces, and shell definition.
  </Step>
</Steps>

## Suite file layout

After a successful suite install you will find the following directory structure:

```
components/
  ui/
    jami-vocabulary.mjs
    jami-ui.css
    jami-primitive-descriptors.mjs
    jami-primitive-components.mjs
    jami-radix-react-wrappers.mjs

studio-ui/
  jami.css
  jami.tailwind.css
  jami-tokens.ts
  jami.shadcn.json
  suites/
    solo.suite.json          ŌåÉ suite manifest
    solo/
      solo-app.implementation.json
      pages/
        solo-today-page.page.json
        solo-today-page.page.implementation.json
        solo-docs-page.page.json
        solo-docs-page.page.implementation.json
      blocks/
        solo-agenda-block.block.json
        solo-agenda-block.block.implementation.json
        ...
```

Each `*.implementation.json` file contains rendered state evidence ŌĆö pre-computed component output for the block's or page's key states (ready, empty, error, long-content, invalid). This evidence is used by the Jami workbench for contract verification and does not affect runtime behaviour.

<Tip>
  Suite items are display-only by design. Action execution, harness runtime, and capability truth all stay in Jami Harness. The installed components show status and surface action references but do not execute actions directly.
</Tip>
