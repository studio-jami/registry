Studio UI is the resident rendering layer of the Jami Studio family. It owns the token system, the primitive vocabulary, the composed components, and the pre-built application workspaces that ship through the public registry.

Studio UI does not own agent execution. Policy decisions, tool side effects, memory writes, provenance, trace emission, and runtime state remain [Jami Harness](/harness/overview) responsibilities. Studio UI renders Harness-emitted payloads through allowlisted components.

## What Studio UI owns

- Design tokens (DTCG source, CSS variables, Tailwind layer, TypeScript types).
- Tokenized primitive components (Button, Panel, Text Field).
- Composed display components (Agent Panel, Data List, Docs Source Panel, Media Grid).
- Selectable brand options for workspace personalities.
- Pre-built application workspaces (Solo, Business Ops, Research Writing, Mixed Media).
- The workbench shell and preview surfaces.

## What Studio UI does not own

- Tool execution.
- Policy and approval decisions.
- Memory writes or retention.
- Audit, trace, and metric emission.
- Run lifecycle, checkpoints, or recovery.

All of those belong to the Harness. Studio UI renders the typed payloads (`uiPayload`, `artifactView`, `actionRef`, `themeRef`, `workspaceRef`) the Harness emits.

## How Studio UI relates to the Registry

The public registry at `registry.jami.studio` distributes Studio UI items via the shadcn protocol. Every Studio UI primitive, component, theme, block, page, app, and workspace has a registry item. Installing an item drops source into your project; no runtime dependency on the registry remains.
